Intercooler 0.4.8 - January 27th, 2015

  • Reworked history support, with the ic-push-url attribute. Renamed the X-IC-SetLocation response header to X-IC-PushURL to be consistent with the new attribute.
  • Include the current URL on all intercooler requests in the ic-current-url parameter.
  • For elements with no text children, apply the fadeFast transition to children only, rather than the parent. This makes buttons look better if their content is wrapped in a span.

Intercooler 0.4.7 - January 5th, 2015

  • Added an integrated visual debugger.
  • Added the 'changed' modifier for ic-trigger-on events
  • Only allow one request in flight at a time for a given element.
  • Allow child elements to trigger parent elements separately.
  • Added ic-trigger-delay attribute to delay firing a request.

Intercooler 0.4.6 - December 30th, 2014

  • Added ic-replace-target attribute, allowing full replacement of target elements for situations like replacing a table row.
  • Added the ability to target elements that are closest to a triggered element in a ic-target attribute via the closest prefix.

Intercooler 0.4.5 - December 12th, 2014

  • Added ic-on-beforeSend, ic-on-success, ic-on-error and ic-on-complete attributes.
  • Always swap content containing a script element, so things are properly reevaluated.

Intercooler 0.4.1 - July 4th, 2014

  • Exposed LeadDyno.processElements() in the Javascript API
  • Begin sending the X-HTTP-Method-Override header up in intercooler requests
  • Begin sending the X-IC-Request header up in intercooler requests

Intercooler 0.4.0 - June 19th, 2014

  • Added the X-IC-Trigger response header to trigger client side event handlers.
  • Added the ic-confirm attribute.
  • Merged a fix for an IE8 compatibility issue.

Comments on 0.4.0 Release

The big enhancement in this release (small code change, but big functionality boost!) is the X-IC-Trigger response header, which finally solves a tricky problem I've been wrestling with: How do you communicate server side state changes that have client-side UI ramifications that fall outside the usual Intercooler request-and-replace partial view flow?

This new mechanism allows you to cleanly separate your server side and client-side logic even in cases where the simple content-swapping approach isn't enough.

A great example is if you want to hide a modal if and only if a form in the modal submits valid data to the server. You can now easily fire an server.accountCreated event (as an example) from the server side and respond to that on the client side by hiding the modal. Clean, crisp and very little code!
